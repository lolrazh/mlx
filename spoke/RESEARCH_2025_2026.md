# Research Synthesis: Training Innovations 2025-2026

> Compiled from 3 Opus/Sonnet research agents scanning arxiv, HuggingFace, blog posts, and technical reports from Chinese AI labs and the broader ML community. Focus: actionable insights for fine-tuning small (1-4B) models on ASR post-processing / text editing tasks on Apple Silicon (24GB).

---

## Tier 1: Immediate Impact (can try this week)

### 1. EPO — Edit-wise Preference Optimization
**Source:** Liang et al., [COLING 2025](https://aclanthology.org/2025.coling-main.229/)

Instead of treating all tokens equally during training, EPO assigns **higher reward to edit tokens** — the tokens that actually change between input and output. Most of our output is identical to input; EPO focuses learning on the few tokens that matter. State-of-the-art on English and Chinese GEC datasets, outperforming standard SFT.

> This is the single most relevant innovation for Spoke. It's the "bounded editing" concept implemented as a training objective. For Spoke, most tokens in the output should be identical to the input — EPO would train the model to focus on the few tokens that need changing (spelling corrections, symbol insertions, etc.).

### 2. Data Repetition Beats Data Scaling
**Source:** [arXiv 2602.11149](https://arxiv.org/abs/2602.11149), February 2026. Code: [GitHub](https://github.com/dkopi/data-repetition)

Under a fixed compute budget, training for **128 epochs on 400 samples dramatically outperformed 1 epoch on 51,200 samples** by 12-26 percentage points. OLMo3-7B trained for 128 epochs on 400 samples outperformed 1 epoch on 51,200 samples. Token accuracy serves as a reliable stopping criterion (NOT val loss — we already knew val loss was unreliable from finding #19).

> We're doing ~6 passes/example at 800 iters. This paper says hundreds of passes could still help. We might be massively undertrained. Consider pushing to 2000+ iters on Qwen3-4B.

---

### 3. LoRA-GA — Gradient-Approximated Initialization
**Source:** Wang et al., [NeurIPS 2024](https://arxiv.org/abs/2407.05000). Code: [GitHub](https://github.com/Outsider565/LoRA-GA)

Initializes LoRA adapters using the eigenvectors of the gradient matrix (via SVD) instead of random initialization. Makes the first LoRA gradient step approximate the first step of full fine-tuning. **2-4x convergence speedup.** +5.69% on GLUE with T5-Base, +11.52% on GSM8K with Llama 2-7B.

> If T11 at 300 iters = 83%, LoRA-GA could potentially reach that in 150 iters, or reach higher accuracy in 300. The SVD initialization step is a one-time cost. Requires implementing custom initialization but the algorithm is straightforward.

---

### 4. rsLoRA — Rank-Stabilized LoRA (One-Line Fix)
**Source:** [Kalajdzievski, HuggingFace](https://huggingface.co/blog/damjan-k/rslora). [arXiv 2312.03732](https://arxiv.org/abs/2312.03732)

Standard LoRA uses scaling factor `alpha/r`, which penalizes higher ranks — they learn slower. rsLoRA changes to `alpha/sqrt(r)`, stabilizing gradients and unlocking the performance benefits of higher ranks.

> We found r=8 = r=16 in our experiments. This might be because standard LoRA's scaling was penalizing our r=16 runs. With rsLoRA, r=16 might actually outperform r=8. Literally a one-line change.

### 5. mlx-lm-lora — DPO/GRPO on Apple Silicon
**Source:** [GitHub: Goekdeniz-Guelmez/mlx-lm-lora](https://github.com/Goekdeniz-Guelmez/mlx-lm-lora)

Community extension to mlx-lm that adds **SFT, DPO, CPO, ORPO, GRPO, Dr-GRPO, DAPO, XPO, PPO**. Custom reward functions via Python decorators. Runs natively on Metal backend. Generates 4 completions per prompt by default (`--group-size 4`).

Example custom reward for Spoke:
```python
from mlx_lm_lora.reward_functions import register_reward_function

@register_reward_function()
def spoke_edit_reward(prompt, completion, reference_answer, **kwargs):
    trigger_score = check_trigger_rules(completion, prompt)
    semantic_score = compute_similarity(completion, reference_answer)
    length_penalty = max(0, (len(completion) - len(prompt) * 1.2)) / len(prompt)
    return 0.5 * trigger_score + 0.4 * semantic_score - 0.1 * length_penalty
```

> This is how we'd do RL training on M4. No CUDA needed.

### 5b. Tool-Call Edit Format (The Coding Agent Approach)
**Source:** [Aider edit formats](https://aider.chat/docs/more/edit-formats.html), [Code Surgery blog](https://fabianhertwig.com/blog/coding-assistants-file-edits/), [Microsoft: Fine-Tuning Small LMs for Function Calling](https://techcommunity.microsoft.com/blog/azure-ai-foundry-blog/fine-tuning-small-language-models-for-function-calling-a-comprehensive-guide/4362539)

Instead of training the model to generate full cleaned text (rewrite), train it to output **structured edit operations** — the same approach coding agents (Aider, Claude Code, Cursor) use for code edits. The model outputs tool calls like `[REPLACE "Whisper" "Wispr"][DELETE ", can you spell that W-I-S-P-R?"]`, and a deterministic function applies edits to the original text.

**Why this matters for Spoke:**
- **No copying errors.** Current approach forces the model to reproduce ~90% unchanged text. Every token is a chance to hallucinate. Edit format only touches what changes.
- **Letter assembly becomes isolated.** Instead of generating "Wispr" mid-sentence, the model outputs `"Wispr"` as a focused replacement value.
- **Fewer output tokens = faster inference.** Most test cases need 1-2 edits (~20 tokens) vs full rewrite (~50-80 tokens).
- **Verifiable.** Can programmatically check if SEARCH string exists in input. Built-in error detection.
- **Small models can learn this.** Microsoft fine-tuned Llama-3.2-3B for function calling. RidgeRun fine-tuned Gemma-3-1B for tool calling. Qwen3 and Llama 3.2 have native tool-calling support in chat templates.

**Format options (need to test which works best):**
```
# Option A: JSON tool calls (native to Qwen3/Llama chat templates)
{"tool": "replace", "find": "Whisper", "replacement": "Wispr"}
{"tool": "delete", "find": ", can you spell that W-I-S-P-R?"}

# Option B: Search/replace blocks (Aider-style, simpler tokens)
<<<REPLACE>>>
Whisper
>>>WITH>>>
Wispr
<<<END>>>

# Option C: Minimal edit notation
REPLACE "Whisper" → "Wispr"
DELETE ", can you spell that W-I-S-P-R?"
```

**Risk:** Schema compliance — will a 3-4B model reliably produce valid structured output? Mitigated by constrained decoding (grammar-based sampling) at inference time. Also: multi-edit ordering and overlapping spans need careful handling.

> This is Phase D ("bounded editing") reframed as tool calling. Connects to EPO (#1) — both focus the model on edit tokens only. The difference: EPO changes the training loss, tool-call changes the output format. Could combine both. **Highest-conviction new idea for breaking past 91%.**

### 5c. Gemma 3 as Fine-Tuning Target
**Source:** [Gemma 3 QAT blog](https://developers.googleblog.com/en/gemma-3-quantized-aware-trained-state-of-the-art-ai-to-consumer-gpus/), [Gemma 3 Distillation Ablations](https://huggingface.co/blog/tawnymanticore/gemma3-ablations)

Gemma 27B has surprised by beating larger models (DeepSeek, Llama 4 Maverick) on multiple benchmarks. The architecture may scale down well. Gemma 3 comes in 1B QAT, 4B, 12B, and 27B sizes. The 4B variant is a direct competitor to Qwen3-4B. Gemma 3 1B QAT is quantization-aware trained from the start — may close the 9% quant gap we see with post-training quantization.

> Fine-tune Gemma 3 4B with same config as T11-ext (all layers, 2000 iters). If the architecture is better at structured tasks, it could beat Qwen3's 91%. Also test Gemma 3 1B QAT as a phone-deployable alternative to LFM2.5-1.2B (70%).

## Tier 2: High Impact, Medium Effort

### 6. RLMEC — RL with Minimum Editing Constraint
**Source:** [ACL 2024 Findings](https://arxiv.org/abs/2401.06081), Renmin University / RUCAIBox. Code: [GitHub](https://github.com/RUCAIBox/RLMEC)

Trains a **generative reward model** to rewrite erroneous outputs with **minimum edit distance**, then uses the token-level comparison between original and minimally-edited versions as fine-grained reward signal. Inspired by the "teacher correcting homework" analogy: identify the incorrect parts, make minimal fixes, grade based on what changed.

> The core idea — reward based on minimum edit distance from the correct output — naturally encourages "preserve A, replace B" behavior. The token-level reward signal focuses learning on the few critical tokens that actually need editing. Combined with mlx-lm-lora's GRPO, this is the RL recipe for Spoke.

### 7. GRPO Variants That Fix Real Problems

#### Dr-GRPO (fixes length bias)
**Source:** [arXiv 2503.20783](https://arxiv.org/abs/2503.20783), COLM 2025

Vanilla GRPO has a **length bias**: longer incorrect responses accumulate disproportionate gradient updates. Dr-GRPO removes length/std normalization terms. Fix is trivial. 43.3% AIME on 7B. Already implemented in mlx-lm-lora.

> Critical for text editing where a 5-token fix and a 200-token rewrite should both get rewarded fairly.

#### DAPO (fixes entropy collapse)
**Source:** [arXiv 2503.14476](https://arxiv.org/abs/2503.14476), ByteDance Seed + Tsinghua

Identifies entropy collapse in GRPO — policy entropy drops, all responses converge to identical outputs. Fixes: Clip-Higher (decouple upper/lower clip), Dynamic Sampling (filter trivially easy/impossible prompts), Token-Level gradients. 50 pts on AIME 2024 vs DeepSeek-R1-Zero's 47, using 50% fewer steps.

> Entropy collapse will hit a 4B model on text editing just as hard. DAPO's Dynamic Sampling is directly relevant: filter our easy/impossible examples from training.

#### GDPO (fixes multi-reward collapse)
**Source:** [arXiv 2601.05242](https://arxiv.org/abs/2601.05242), January 2026

When combining multiple reward signals, GRPO sums then normalizes → distinct rewards collapse. GDPO normalizes each reward independently before aggregation. Drop-in replacement.

> Perfect for Spoke: format compliance + content accuracy + preservation fidelity = three rewards that shouldn't be collapsed.

#### GTPO (token-level reward weighting)
**Source:** [arXiv 2508.04349](https://arxiv.org/abs/2508.04349), August 2025

Uses the model's own policy entropy at each token position as a proxy for "cognitive effort." High-entropy tokens in correct sequences get rewarded more. +60% over DAPO on AIME 2025.

> In text editing, the "hard" tokens are the edit decision points. GTPO naturally gives more signal to those vs trivially-copied surrounding text.

### 8. Distillation from Large Teacher (The Proven Recipe)
**Source:** Qwen3 Technical Report, DeepSeek R1, SmolLM3

Both Qwen and DeepSeek **skip RL entirely for sub-4B models**. They use strong-to-weak distillation:
- Generate high-quality outputs from a 32B+ teacher model
- SFT the small model on those outputs
- The teacher's outputs ARE the training signal

Qwen3 small models (0.6B-4B) use two-phase distillation:
1. **Off-policy distillation**: Teacher generates outputs, student learns from them
2. **On-policy distillation**: Student generates own responses, uses them for self-distillation

SmolLM3's recipe: Qwen3-32B generates reasoning chains from non-reasoning datasets. APO (Anchored Preference Optimization) for alignment — more stable than DPO for small models.

> For Spoke: use Claude Opus or Qwen3-32B API to generate 3-5K examples across all 9 triggers. More effective than RL at our scale. This is the highest-leverage move.

---

### 9. GRAPE — Select Training Data That Fits Your Model
**Source:** Zhang et al., [NeurIPS 2025 Spotlight](https://arxiv.org/abs/2502.04194)

For each training example, selects the response that aligns most closely with the target model's pretrained distribution (measured by normalized probability). Data too far from what the model "expects" hurts more than it helps. With 1/3 the data and half the epochs, GRAPE enables LLaMA-1-8B to exceed Tulu3-SFT by 3.5%.

> Run base Qwen3-4B on all 535 examples, compute log-likelihood of target outputs, rank by likelihood. Train on top 80%. Simple, cheap filter that could remove harmful examples.

### 10. Synthetic Data Generation Techniques

#### CRAFT (Corpus Retrieval and Augmentation for Fine-Tuning)
**Source:** [TACL / EMNLP 2025 Oral](https://arxiv.org/abs/2409.02098). Code: [GitHub](https://github.com/ziegler-ingo/CRAFT)

Retrieves relevant real documents via similarity search, then restructures them into task-formatted training samples. Outperforms Self-Instruct and Evol-Instruct. CRAFT-generated data beats human-curated data by 46 preference points on summarization.

> Find real ASR transcript examples from public corpora with speech-like patterns, reformat into Spoke input/output pairs.

#### Instruction Reversal (Phi-4)
**Source:** [Microsoft Phi-4 Technical Report](https://www.microsoft.com/en-us/research/publication/phi-4-technical-report/)

Take correctly formatted outputs and reverse-engineer the corresponding "messy" inputs. Gives clean input-output pairs where you control quality of the target.

> Take correctly formatted ASR outputs, generate messy inputs. Cheap way to double training data.

#### LANCE (Self-Evolving Data)
**Source:** [EMNLP 2025](https://aclanthology.org/2025.emnlp-main.914/). Code: [GitHub](https://github.com/Control-derek/LANCE)

Model generates instruction and preference data guided by a Constitution (set of rules), then fine-tunes on its own output in cycles. No external teacher needed. +3.64 average score for Qwen2-7B.

> Seed with 535 Spoke examples, have model generate new editing examples using constitutional rules ("edits must be minimal," "preserve speaker intent"), filter and retrain.

#### DeepSeek R1 Rejection Sampling
**Source:** [DeepSeek-R1](https://arxiv.org/abs/2501.12948)

Generate 10 candidates per input with current model, score against ground truth (edit distance, exact match), keep only the best, retrain. Self-distillation that works at any model size.

---

### 11. Token Cleaning — Token-Level Quality Filter
**Source:** Pang et al., [ICML 2025](https://arxiv.org/abs/2502.01968). Code: [GitHub](https://github.com/UCSC-REAL/TokenCleaning)

Evaluates quality at the individual token level, not whole samples. Measures each token's influence on model updates and masks out low-value tokens during training.

> In our training pairs, unchanged tokens (most of the text) are arguably noise — the model should focus on edited spans. Token Cleaning automatically upweights edit tokens, similar to EPO's philosophy.

## Tier 3: Strategic (changes architecture or pipeline)

### 12. Microsoft Mu — Cross-Architecture Encoder-Decoder Distillation
**Source:** [Microsoft](https://github.com/microsoft/encoder-decoder-slm). Deployed in Windows Settings on Copilot+ PCs.

330M encoder-decoder distilled from Phi. Uses 2/3 encoder, 1/3 decoder split. +6 ROUGE-L from cross-architecture distillation. 47% lower first-token latency and 4.7x higher throughput vs decoder-only at same param count.

> Validates LEDGER finding #25 (decoder-only is wrong for editing). If we go encoder-decoder, Mu's recipe is the blueprint. Train large decoder-only → distill into small encoder-decoder.

---

### 13. QAT During Fine-Tuning (Close the Quant Gap)
**Source:** [Google Gemma 3 QAT](https://developers.googleblog.com/en/gemma-3-quantized-aware-trained-state-of-the-art-ai-to-consumer-gpus/), [L4Q (ACL 2025)](https://aclanthology.org/2025.acl-long.99/)

Google's recipe: ~5,000 additional training steps with fake quantization ops. Uses non-quantized checkpoint's logits as targets (self-distillation under simulated int4). Result: **54% reduction in perplexity degradation** vs post-training quantization.

L4Q combines LoRA + QAT in a single pass. Memory overhead comparable to plain LoRA. Works at 3-bit and 4-bit.

> Could close our 9% quant gap (bf16→6-bit). But needs Unsloth (not native MLX yet). Alternative: train with `--use-quantized-base` in mlx-lm (feature exists but not well documented).

---

### 14. VerIF — Verification Engineering for Instruction Following
**Source:** [arXiv 2506.09942](https://arxiv.org/abs/2506.09942), Tsinghua KEG Lab, EMNLP 2025

Combines **rule-based code verification** for hard constraints (length, format, structure) with **LLM-based verification** for soft constraints (semantic correctness). Comes with 22k-instance IF dataset.

> Our 9 Spoke triggers are natural rubrics. Rule-based checks for "did it fix the spelling?" + LLM judge for "does it sound right?" This is the reward design for GRPO.

---

### 15. G2RPO-A — Guided GRPO for Small Models
**Source:** [arXiv 2508.13023](https://arxiv.org/abs/2508.13023). Code: [GitHub](https://github.com/T-Lab-CUHKSZ/G2RPO-A)

GRPO's success depends on strong world knowledge. Small models (sub-4B) lack this. G2RPO-A injects **ground-truth reasoning steps** into rollout trajectories, with adaptive reduction of guidance as model improves.

> Specifically designed for "small model + RL." For text editing, "ground-truth reasoning steps" = injecting the correct edit rationale into rollouts.

---

### 16. REDI — Reinforcement Distillation
**Source:** [arXiv 2505.24850](https://arxiv.org/abs/2505.24850), May 2025

Extends DeepSeek's distillation by using **both positive AND negative** teacher traces. Standard distillation throws away wrong answers; REDI uses them as negative signal. Qwen-REDI-1.5B matches DeepSeek-R1-Distill-Qwen-1.5B on MATH-500.

> We already have bad examples (model failures on test cases). REDI shows you can use those failures as training signal, not just discard them.

## The Pipeline That Emerges

Based on all research, here's the state-of-the-art recipe for our task:

```
Current:   SFT (LoRA) → post-training quantize → deploy
                ↓
Optimal:
  1. DISTILL 3-5K examples from large teacher (Claude/Qwen3-32B)
     - Use 9 trigger categories as seeds
     - Instruction reversal: take clean outputs, generate messy inputs
     - GRAPE filter: keep only examples that fit model's distribution

  2. SFT with LoRA improvements
     - LoRA-GA initialization (2-4x faster convergence)
     - rsLoRA scaling (unlock higher ranks)
     - Many more epochs (data repetition paper says 100+ epochs still help)
     - Token accuracy as stopping criterion, NOT val loss

  3. DPO/EPO with edit-weighted token rewards (mlx-lm-lora)
     - EPO: upweight edit tokens, downweight pass-through tokens
     - Use current model's failures as negative examples (REDI)

  4. GRPO with spoke_reward() function (Dr-GRPO variant via mlx-lm-lora)
     - Rule-based: trigger corrections happened? (VerIF-style)
     - Edit distance: how close to reference? (RLMEC-style)
     - Preservation: did it keep unchanged text? (GDPO multi-reward)

  5. QAT quantization (when tooling supports it on MLX)
```

---

## The Single Most Important Insight

**Chinese labs don't use RL for small models.** Qwen, DeepSeek, and SmolLM all use **distillation from a large teacher** for sub-4B. The teacher's outputs ARE the training signal. RL is only added at 7B+ where the model has enough world knowledge for self-improvement.

For our 4B model on 535 examples, **better data from a stronger teacher** is the highest-leverage move — not fancy training algorithms. Generate 3-5K high-quality examples from Claude Opus or Qwen3-32B, and a single SFT pass will likely surpass anything we can achieve by tuning hyperparameters on 535 examples.
